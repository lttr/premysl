The big picture

One snapshot, two ways to search it

Linking and refreshing a repository works exactly as before and produces the same snapshot. From there the road forks: a chat searches that snapshot either with grep retrieval or with RAG retrieval — never both — decided by the chat's retrieval mode.

Because both paths end in the same source cards, what you compare is the quality of what was retrieved — not two different interfaces.

Why this exists

Same question, different words

Grep matches the words you type. When your docs phrase an idea differently, grep finds nothing. RAG matches meaning, so it still surfaces the right passage.

The one new control

Each chat commits to a method

When you start a chat you pick its retrieval mode. It sits next to the model picker but is a separate choice — so you can hold the model fixed and vary only the retrieval method. The mode is shown while you chat and stays fixed for the chat's life.

Behavior · behind the scenes

What happens when you link or refresh

RAG needs the docs prepared ahead of time. That preparation is bolted onto the existing link/refresh step — it runs while you wait, in the same request, with no background jobs. Grep needs none of this, so if you never use RAG, none of it runs.

Snapshot written
markdown on disk
→
Chunk each file
by section, with fallback
→
Embed chunks
Voyage · grouped per file
→
Store rows
vector + text + FTS5
All-or-nothing. A repo is indexed completely or not at all — chunk rows are written in one transaction. If embedding fails partway (timeout, provider error), the repo is left clean and you simply refresh to retry. Files embed a few at a time so it finishes inside the one request.

Refresh

Rebuilds the chunk rows from the new snapshot, so RAG is exactly as fresh as grep — staleness can't skew the comparison. (Full rebuild today; re-embedding only changed files is planned future work.)

Unlink

Deletes the chunk rows together with the snapshot folder. Nothing is left behind.

Behavior · preparing the docs

How a file becomes chunks

A document is cut into passages the assistant can retrieve precisely. Where there are headings, each section is one chunk. Where there are none (a bare file, or the text before the first heading), it falls back to fixed-size windows — so every file is covered, not just tidy ones.

heading-based chunks
1# Refresh
2A snapshot is frozen…
3it drifts until refreshed.
4
5## Cadence
6Manual and on-demand only.

Each chunk carries its heading path (“Refresh › Cadence”) and its line range, for accurate citations.

window fallback (no headings)
1raw notes, no structure…
2more loose lines…
3…cut at ~500 tokens
4…~50 token overlap…
5next window begins

Voyage embeds these in the context of their file, so even structureless chunks get usable signal.

Behavior · answering a RAG question

From question to three chunks

Two independent rankings are produced — one by meaning, one by keywords — then blended into a single order. The top three chunks become the source cards.

Top 3 chunks

Blending both rankings is why RAG stays strong on exact terms (an error code, a name) while also catching paraphrase — the keyword half doesn't disappear, it votes alongside meaning.

RAG returns a few precise chunks (top 3), not whole files. Grep keeps its whole-file behavior. That's a deliberate difference in what each method hands to the assistant.
Architecture · where things live

One database does both halves

The chunks, their meaning-vectors, and the keyword index all live in the app's existing SQLite database — no extra service, no special extension. The database itself does the meaning math and the keyword scoring; the app only blends the two lists.

chunk rows (one table)
  • text — the passage
  • vector(1024) — its meaning, from Voyage
  • repo · path · lines — for the citation
built-in database features
  • Vector distance → the meaning ranking
  • Full-text (BM25) → the keyword ranking
  • App blends the two → final order

We confirmed live that the production database (libSQL) supports both natively, so there's no new infrastructure and no fragile add-on to deploy.
