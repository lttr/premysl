Linked Repository Retrieval

Link your own GitHub repos as sources, and the assistant searches their text — your docs and notes — to ground and cite its answers. This page shows the key screens, the behaviour behind them, and how the pieces fit together. It's for shared understanding, not implementation detail.

00 · Orientation

The whole thing in one breath

Four moving parts. Everything below is just a closer look at one of them.

🔗 Link

Open a picker from the sidebar, search your own repos, and link the ones you want. Linking immediately downloads a snapshot of the repo's files to the server.

🔍 Retrieve

Ask a question in chat. The assistant text-searches every linked snapshot in one step and pulls back the relevant pieces of files as grounded, cited context.

🔄 Refresh

Snapshots are frozen in time. When a repo changes on GitHub, you press refresh to re-download it. Nothing updates on its own.

What each row tells the owner

The list answers one question at a glance: what does the assistant currently have access to, and how fresh is it?

Repo name The full owner/name — the identity that keys both the link record and the on-disk snapshot.
“refreshed Xh ago” Last-refreshed time, so the owner can judge whether results are current before trusting them.
↻ Refresh Re-downloads the default branch and replaces the snapshot. Spins while running.
✕ Unlink Removes the repo as a source and deletes its snapshot from disk. Asks to confirm first.
＋ on the header Opens the repository picker (next section).
Freshness is honest. The timestamp exists precisely because snapshots never update themselves — the owner is never surprised by stale results after a push.

Three row states

  • Link — not yet a source. One click links it and starts the snapshot download.
  • Linking… — the snapshot is downloading right now. It becomes searchable the moment it lands.
  • Linked ✓ — already a source. Shown disabled so the same repo can't be added twice.

Why owner-only, and why client-side search

The list is filtered to repos the owner owns so it matches their mental model of “my repos” and stays short. Because it's short, the full list is fetched once and filtered in the browser as they type — no round-trip per keystroke. Server-side search is only revisited if the list grows into the hundreds.

Anatomy of a snippet

Every match is a self-contained, citable unit. Provenance on top, content below.

Repository Which linked repo the match came from — the owner searches all of them at once and never names one.
File path Path inside that repo, so the answer can cite it and the owner can open the source.
Location The matched line range — a window around the hit, not just the single line. “Open” links to that range on GitHub, pinned to the snapshot's commit so it stays accurate after later pushes.
The piece A whole small file, or a window of lines — coherent enough to actually answer from.

Nothing found

No matches in your linked repositories for “roll back drizzle migration”. Try rephrasing, or link another repository.
Honest empties matter. The assistant says when it found nothing, so the owner knows to rephrase or link a different repo — rather than getting a confident answer with no grounding.
04 · Behaviour

What happens when you link a repository

One click in the picker sets off a short chain. By the end, a snapshot is on disk and immediately searchable — no extra step.

L Owner
1 Click Link on a repo in the picker
Server
2 Verify it's the owner (locked mode gate)
→
3 Record the link in linkedRepositories
GitHub
4 Download the default-branch tarball using the session token
Disk (.data)
5 Extract markdown only into .data/repos/<id>/
→
6 Row flips to Linked ✓ — now searchable
No background work, ever. Every GitHub call happens inside a request the owner made, using the token that lives only in their session. There is no webhook, no scheduler, no offline job — by design.
05 · Behaviour

What happens when you ask a question

The assistant decides a question needs the owner's material, calls one retrieval tool with a plain query, and gets back ranked pieces of files. No repository argument — it searches them all.

L Owner
1 Asks a question in chat
P Assistant
2 Decides it needs the owner's material → calls the retrieval tool with a free-text query
Snapshots
3 ripgrep across all of this owner's snapshot directories
→
4 For each hit, take the whole file (if small) or a window of lines
P Assistant
5 Receives snippets with repo · path · location
→
6 Renders the result card & answers with citations
One query, all repos. The tool takes only a query — no “which repo?”. The owner never has to know where their answer lives.
Pieces, not lines. Returning a coherent chunk (small whole file, or a line window) is what makes the context good enough to answer from. Markdown-section-aware extraction is a later refinement.
06 · Behaviour

Refresh and unlink

Two small lifecycles that keep snapshots current and let the owner take a repo out of scope.

Refresh a snapshot

Owner-initiated, manual only.

  1. Owner presses ↻ on a linked-repo row.
  2. Server re-downloads the current default-branch tarball.
  3. It replaces the old snapshot on disk wholesale.
  4. The last-refreshed timestamp updates → row shows “just now”.
Nothing auto-updates. Pushing to GitHub does not change a snapshot. Until the owner refreshes, searches reflect the last refresh — which the timestamp always shows.

Unlink a repository

Removes it as a source, cleanly.

  1. Owner presses ✕ and confirms.
  2. The linkedRepositories row is deleted.
  3. The on-disk snapshot directory is removed.
  4. Retrieval no longer searches it — it's fully out of scope.
Leaves nothing behind. Both the record and the files are gone, so an unlinked repo can never resurface in a search result.
07 · Architecture

How the pieces fit together

Three places things live: the browser, the server, and GitHub. The owner's GitHub token rides along in their session only; snapshots live on the persistent disk volume.

🖥️ Browser app/

  • Sidebar: linked-repo list + ＋
  • Repository picker (live filter)
  • Chat + retrieval result card
request

⚙️ Server server/

  • Owner gate on every handler
  • Link · unlink · refresh · list
  • Per-request retrieval tool (ripgrep)
  • Session holds the GitHub token
session token

🐙 GitHub api

  • List the owner's repos
  • Download default-branch tarballs (markdown kept, history dropped)
  • Read-only — no writes ever
🔐 The token

GitHub login is extended to request repo scope. The token is kept in the session's server-only field — never sent to the client, never written to the database.

💾 The snapshots

Markdown files extracted from the tarball under .data/repos/<id>/ — shallow, no git history, kept small. The .data volume is mounted in production, so snapshots survive redeployment — no re-linking after a deploy.

🗄️ The records

One linkedRepositories row per link in SQLite, keyed to the owner. Cascade-deletes with the user; the snapshot directory is the on-disk half of the same record.

Why download whole repos instead of GitHub's code-search API? That API is default-branch only, caps files at 384 KB, allows ~9 requests/minute, and matches prose weakly — a poor fit for a docs assistant. Local snapshots dodge all four, and the extracted markdown is also the natural input for the future RAG phase.
08 · Data

09 · Boundaries

What phase 1 is — and isn't

The deliberate line. Everything on the right is a known, intentional deferral.

✓ In phase 1

  • Link / unlink repos you own (public & private)
  • Snapshot on link; manual refresh
  • Markdown only; shallow, history-free snapshots
  • Plain text search (ripgrep) over all snapshots
  • Whole small files or a line window per match
  • Commit-pinned, citable results rendered inline in chat
  • Owner-only in locked mode; token in-session only
  • Snapshots survive redeployment

— Deferred / out of scope

  • RAG / semantic search (embeddings) — phase 2
  • Collaborator & organisation repos
  • Non-markdown files (code, data, binaries)
  • Automatic, scheduled, or webhook refresh
  • Persisting the GitHub token at rest
  • Writing to repos (issues, PRs, commits)
  • Branch selection beyond the default branch
  • Per-chat repo scoping (links are account-level)
  • Markdown-section-aware extraction
10 · Language

Words we use exactly

Canonical glossary terms. Use these, not their aliases.

GitHub connection
The owner's logged-in GitHub access, carried in their session — what lets the server call GitHub on their behalf.
Linked repository
A repo the owner has added as a source. The act is linking.
Snapshot
The downloaded, on-disk copy of a repo's files that retrieval searches.
Refresh
Re-downloading a snapshot to replace it with current contents. Always manual.
Owner
The single account (NUXT_OWNER_GITHUB_HANDLE) that, in locked mode, may link and retrieve.
